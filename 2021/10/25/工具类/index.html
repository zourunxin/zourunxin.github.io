Git

下载

  1. 下载默认分支：git clone "ssh 地址"，如git clone git@code.byted.org:lijing.owen/test-demo.git
  2. 下载指定分支：git clone --branch 7.9 --single-branch https://github.com/elastic/logstash.git；–branch 后是分支名
  3. 将远程分支更新到本地所在分支：
    • 默认远程同名分支：git pull
    • 指定远程分支：git pull origin master（master 为远程分支）
  4. 将远程分支更新到本地指定分支：git pull (--force) origin master:master（第一个 master 为远程分支，第二个为本地已有分支）
  5. 将远程分支下载到本地新分支（本地分支将被新建）：git fetch origin master:zourx（将远程 master 分支下载到本地 zourx 分支）

上传

  1. 将本地文件夹初始化为 git 可管理的仓库：git init；
  2. 将本地仓库与远程仓库关联：git remote add origin ssh，如git remote add origin git@code.byted.org:ad/addrd-stats-thanos.git
  3. 将文件添加到本地仓库：git add . （. 意味着整个项目的文件添加到仓库，若将 . 改为指定文件名，则添加指定文件到本地仓库）
  4. 评论此次仓库更新的内容：git commit -m "对文件的评注"
  5. 将本地仓库的内容推送到远程：git push origin 本地分支:远程分支，如git push origin master:feature/20210915_add_game_dim。

删除

  1. 删除本地分支：git branch -d 分支名，如git branch -d zourx
  2. 删除远程分支：git push origin --delete 远程分支名，如git push origin --delete zourx。

对本地分支的操作

  1. 查看冲突（查看当前分支与本地 master 分支的冲突）：git rebase master（在 idea 的 git 工具的 resolve conflicts 可查看发生冲突的文件。）
  2. 比较当前分支与本地 master 分支：git diff master
  3. 合并 master 分支到当前分支：git merge master
  4. 将暂存区和工作区都回退到上一次版本，并删除所有修改信息：git reset --hard

Tensorflow

1. tf.keras 搭建网络八股（Sequential）

1.1 六步法

  1. import
  2. train, test —— 准备训练集和测试集（区分输入特征和标签）
  3. model = tf.keras.models.Sequential —— 在 Sequential() 搭建网络结构，逐层描述每层结构
  4. model.compile —— 在 compile() 配置训练方法，选择优化器、损失函数、评测指标
  5. model.fit —— 在 fit() 中执行训练过程，告知训练集和测试集的输入特征和标签、告知 batch 大小（样本大小），epoch 大小（迭代次数）
  6. model.summary —— 使用 summary() 打印网络的结构和参数统计

1.2 举例

1.2.1 Sequential()举例

model = tf.keras.models.Sequential ([ 网络结构 ]) #描述各层网络（从输入层到输出层每一层的网络结构）

网络结构举例：

  • 拉直层：tf.keras.layers.Flatten() —— 把输入特征拉直变成一维数组

  • 全连接层： tf.keras.layers.Dense(神经元个数, activation= “激活函数“ ,kernel_regularizer=哪种正则化)

    • activation（字符串给出）可选: relu、 softmax、 sigmoid 、 tanh
    • kernel_regularizer 可选: tf.keras.regularizers.l1()、tf.keras.regularizers.l2()

  • 卷积层： tf.keras.layers.Conv2D(filters = 卷积核个数, kernel_size = 卷积核尺寸 strides = 卷积步长， padding = “ valid” or “same”)

  • LSTM层（循环神经网络层）： tf.keras.layers.LSTM()

1.2.2 compile()举例

model.compile(optimizer = 优化器, loss = 损失函数, metrics = [“准确率”] )

  • Optimizer可选:
    ‘sgd’ or tf.keras.optimizers.SGD (lr=学习率,momentum=动量参数)
    ‘adagrad’ or tf.keras.optimizers.Adagrad (lr=学习率)
    ‘adadelta’ or tf.keras.optimizers.Adadelta (lr=学习率)
    ‘adam’ or tf.keras.optimizers.Adam (lr=学习率, beta_1=0.9, beta_2=0.999)

  • loss可选:
    ‘mse’ or tf.keras.losses.MeanSquaredError()
    ‘sparse_categorical_crossentropy’ or tf.keras.losses.SparseCategoricalCrossentropy(from_logits=False)

  • Metrics可选:
    ‘accuracy’ ：y_和y都是数值，如y_=[1] y=[1]
    ‘categorical_accuracy’ ：y_和y都是独热码(概率分布)，如y_=[0,1,0] y=[0.256,0.695,0.048]
    ‘sparse_categorical_accuracy’ ：y_是数值，y是独热码(概率分布),如y_=[1] y=[0.256,0.695,0.048]（重要）

1.2.3 fit()举例

model.fit(训练集的输入特征, 训练集的标签, batch_size= , epochs= , validation_data=(测试集的输入特征，测试集的标签), validation_split=从训练集划分多少比例给测试集，validation_freq = 多少次epoch测试一次)

  • batch_size：样本数
  • epochs：数据集迭代次数
  • validation_data 和 validation_split 两者选一
  • validation_freq：多少次 epochs 迭代使用测试集验证一次结果

4. 知识点

  1. 对输入网络的输入特征进行归一化处理，即使输入特征值保持在 0 - 1 之间，把输入特征的数值变小更适合神经网络吸收。
  2. 对于图上像素点来说，纯黑色用 0 表示，纯白色用 255 表示
  3. tf.reshape(tensor, shape = [2, 3, 3])，表示将输入的 tensor 的格式转化为 2 * 3 * 3 的数组大小（三维），如果 shape 列表仅两个元素，则转化为二维数组形式。
  4. tf.Tensor 是一种格式，里面的 shape()指示了数据的维度（如 2 * 3 形式的二维数组，则为 shape(2, 3)
  5. tf.constant()，将输入的数字、一维列表、二维列表转化为 tf.tensor 格式
  6. axis = 0 代表横向，axis = 1 代表纵向。但在计算时，需要注意：
  1. x_train.reshape( , , , )，当其中有四个参数时，依次分别是：样本数量、行数、列数、通道数
  2. tf.one_hot(indices, depth, on_value=None, off_value=None, axis=None, dtype=None, name=None)，one_hot 编码 API。
    • indices 是任意维数组，输出的结果为 indices 的（维度 + 1）维数组。
    • depth 为要扩展的维度大小
    • on_value 索引位置处的值，默认为 1
    • off_value 非索引位置处的值，默认为 0
    • axis 指定扩展的维度，默认为 -1，横向扩展维度
    • dtype 要输出的值的类型
    • name 选填
    需要注意：tf.one_hot()方法不适用元素为 float 类型，故在调用该方法前，需使用 x = tf.cast(x, dtype=int32)将其指定为 int 类型
  3. 一维卷积及参数解析
    • filters：卷积核个数
    • kernel_size：卷积核大小，一维卷积只宽的大小
    • strides：卷积的步长
    • padding：”valid”, “causal” 或 “same” 之一 (大小写敏感)。”valid” 表示「不填充」。 “same” 表示填充输入以使输出具有与原始输入相同的长度。 “causal” 表示因果（膨胀）卷积。
    • data_format: 字符串, “channels_last” (默认) 或 “channels_first” 之一，表示输入的各个维度顺序。 “channels_last” 对应输入尺寸为 (batch, steps, channels) (Keras 中时序数据的默认格式) 而 “channels_first” 对应输入尺寸为 (batch, channels, steps)。
    • dilation_rate: 一个整数，或者单个整数表示的元组或列表，指定用于膨胀卷积的膨胀率。当前，指定任何 dilation_rate 值 != 1 与指定 stride 值 != 1 两者不兼容。
    • activation: 要使用的激活函数。如未指定，则不使用激活函数(即线性激活：a(x) = x)。
    • use_bias: 布尔值，该层是否使用偏置向量。
    • kernel_initializer: kernel 权值矩阵的初始化器 (详见 initializers)。
    • bias_initializer: 偏置向量的初始化器 (详见 initializers)。
    • kernel_regularizer: 运用到 kernel 权值矩阵的正则化函数 (详见 regularizer)。
    • bias_regularizer: 运用到偏置向量的正则化函数 (详见 regularizer)。
    • activity_regularizer: 运用到层输出（它的激活值）的正则化函数 (详见 regularizer)。
    • kernel_constraint: 运用到 kernel 权值矩阵的约束函数 (详见 constraints)。
    • bias_constraint: 运用到偏置向量的约束函数 (详见 constraints)。

输入尺寸：3D 张量 ，尺寸为 (batch_size, steps, input_dim)。
输出尺寸：3D 张量，尺寸为 (batch_size, new_steps, filters)。由于填充或窗口按步长滑动，steps 值可能已更改。
补充：tf.keras.layers.Conv1D(32, 3, activation=’relu’, input_shape=input_shape[2:])(x)

    • kernel_size: 一个整数，或者 2 个整数表示的元组或列表， 指明 2D 卷积窗口的宽度和高度。 可以是一个整数，为所有空间维度指定相同的值。
    • strides: 一个整数，或者 2 个整数表示的元组或列表， 指明卷积沿宽度和高度方向的步长。 可以是一个整数，为所有空间维度指定相同的值。 指定任何 stride 值 != 1 与指定 dilation_rate 值 != 1 两者不兼容。
    • padding: “valid” 或 “same” (大小写敏感)。
    • data_format: 字符串， channels_last (默认) 或 channels_first 之一，表示输入中维度的顺序。 channels_last 对应输入尺寸为 (batch, height, width, channels)， channels_first 对应输入尺寸为 (batch, channels, height, width)。 它默认为从 Keras 配置文件 ~/.keras/keras.json 中 找到的 image_data_format 值。 如果你从未设置它，将使用 channels_last。

    • activation: 要使用的激活函数 (详见 activations)。 如果你不指定，则不使用激活函数 (即线性激活： a(x) = x)。
    • use_bias: 布尔值，该层是否使用偏置向量。
    • kernel_initializer: kernel 权值矩阵的初始化器 (详见 initializers)。

    • bias_regularizer: 运用到偏置向量的正则化函数 (详见 regularizer)。

    • kernel_constraint: 运用到 kernel 权值矩阵的约束函数 (详见 constraints)。
    • bias_constraint: 运用到偏置向量的约束函数 (详见 constraints)。

Kafka

Kafka 是什么

Kafka 是一个分布式流式处理平台，拥有高吞吐（但不支持 topic 消息有序）、可持久化、可水平扩展、支持流数据处理等特性。
整体可划分三个模块：

  1. 消息系统
  2. 存储系统
  3. 流式处理平台

典型结构
一个典型的 kafka 体系架构包括若干 producer、若干 broker、若干 consumer，以及一个 ZooKeeper 集群。

ZooKeeper

ZooKeeper 负责 Kafka 集群元数据的管理、控制器的选举等操作。

参数 含义 默认值 是否必填 备注
bootstrap.servers 指定生产者客户端连接 kafka 集群所需的 broker 地址清单，如 host1:port1, host2:port2 “” 是 并非需要所有 broker 的地址，但设置至少两个及以上

client.id 设定 KafkaProducer 对应的客户端 id “” 否 若不填，KafkaProducer 会自动生成形如“producer-1“的非空字符串
acks 指定分区中必须要有多少个副本收到这条消息，之后生产者才认为这条消息是成功写入的 1 否 acks 参数类型需是字符串。可为 1、0、(-1 和 all)（都是等待所有副本成功写入才能收到服务端的响应
max.request.size 指定生产者客户端能发生的消息的最大值 1048576B，即 1MB 一般来说，默认值即足够满足多数场景，若要修改该参数，需同时修改 broker 端的 message.max.bytes 参数
retries 用来配置生产者重试的次数 0 默认在发生异常的时候不进行任何重试动作。重试可以解决如网络抖动、leader 副本选举带来的异常
retry.backoff.ms 设定两次重试之间的时间间隔 100 与 retries 参数搭配使用。需注意不能设置太小导致无效的频繁重试，不能设置太大导致生产者过早放弃重试
compression.type 指定消息的压缩方式 none 该参数还可以配置为“gzip”、“snappy”和“lz4”。消息压缩是一种使用时间换空间的优化方式
connections.max.idle.ms 指定在多久之后关闭限制的连接 540000ms，即 9min
linger.ms 指定生产者发生 ProducerBatch 之前等待更多消息（ProducerRecord）加入 ProducerBatch 的时间 0 生产者客户端会在 ProducerBatch 被填满或等待时间超过 linger.ms 值时发生出去。增大这个参数会增加消息的延迟，但能提升一定的吞吐量，该参数与 TCP 中的 Nagle 算法有异曲同工之妙
receive.buffer.bytes 设置 Socket 接收消息缓冲区（SO_RECBUF)的大小 32768B，即 32KB 若该值设为 -1，则使用操作系统的默认值
send.buffer.bytes 设置 Socket 发送消息缓冲区（SO_SNDBUF)的大小 131072B，即 128KB 若该值设为 -1，则使用操作系统的默认值
request.timeout.ms 配置 Producer 等待请求响应的最长时间 30000(ms) 请求超时之后可以选择进行重试。该参数需要比 broker 端参数 replica.lag.time.max.ms 值要大，这样可以减少因客户端重试而引起的消息重复的概率

补充：

  • KafkaProducer 是线程安全的，可以在多个线程中共享单个 KafkaProducer 实例，也可以将 KafkaProducer 实例进行池化供其他线程调用。
  • 若在创建 KafkaProducer 实例时没有设定 key/value.serializer 这两个配置参数，那么就需要在构造方法中添加对应的序列化器。如KafkaProducer<String, String> producer = new KafkaProducer<>(props, new StringSerializer(), new StringSerializer());

Kafka 基础概念

主题与分区

主题（topic）是逻辑上的概念，分区（partition）在存储层面可看作一个可追加的日志文件。一个 topic 包含一个或多个 partition，但一个 partition 只属于单个 topic。
消息在写入某个 topic 时，实际上写入的是 topic 的某个 partition（根据分区规则决定）。消息在被追加到日志文件的时候都会被分配一个特定的偏移量（offset），用来保证消息在分区的顺序性。Kafka 保证分区有序而不是主题有序。分区规则有如下四种：

  1. specify partition
  2. random
  3. key hash
  4. custom

多副本机制

Kafka 通过多副本机制实现了故障的自动转移，当 Kafka 集群中某个 broker 失效时仍能保证服务可用。
通过增加副本数量以提升容灾能力，即一个分区有多个副本。有如下特点：

  • 同一分区的不同副本中保存的是相同的消息（但在同一时刻，副本之间并非完全一样），副本之间是“一主多从”关系。
  • leader 副本负责处理读写请求，follower 副本只负责与 leader 副本的消息同步。
  • 副本处于不同的 broker 中，当 leader 副本出现故障时，Zookeeper 会从 follower 副本中重新选举新的 leader 副本以继续对外提供服务。
  • 分区中的所有副本统称为 AR(Assigned Replicas)。所有与 leader 副本保持一定程度同步的副本（包括 leader 副本在内）组成 ISR(In-Sync-Replicas)，而与 leader 副本同步滞后过多的副本（不包括 leader 副本）组成 OSR(Out-of-Sync Replicas)。这个同步滞后的范围可以通过参数进行配置。
  • AR = ISR + OSR。
  • leader 副本负责维护和跟踪 ISR 集合中所有 follower 副本的滞后状态。当 follower 副本落后太多或失效时，leader 副本会将其从 ISR 集合中剔除；如果 OSR 集合中有副本“追上”了 leader 副本，则 leader 副本会将它从 OSR 集合转移至 ISR 集合。
  • 默认情况下，当 leader 副本出现故障时，只有 ISR 集合中的副本才有资格被选举为新的 leader（这个规则可以通过修改参数来改变）。
  • HW(High Watermark)：高水位，由该 partiton

特性与实现

  • 负载均衡：同一 topic 的不同消息分配到不同 partition，不同 partition 分配到不同机器。
  • 可靠性：同一 partition 的不同副本分配到不同的机器
  • 数据一致性：副本区分 leader 和 follower，producer 和 consumer 只和 leader 进行交互，follower 则不停地从 leader 同步数据。
